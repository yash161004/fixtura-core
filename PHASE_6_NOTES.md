# Phase 6 Notes: End-to-End Test

## 1. Implementation Summary

Phase 6 validates the entire lifecycle of an AI agent's execution within Fixtura: from live recording, to passive offline replay, interactive inspection, automatic grading, and visual rendering. 

- **Interactive Inspector Graceful Termination (`cli/inspect.py`)**: 
  - Catches `EOFError` properly during standard input (`input()`) queries. 
  - This is critical for automated pipelines. By piping an empty string (`echo "" | fixtura inspect`), the loop naturally finishes all steps and exits cleanly with a 0 status code, rather than throwing a blocking Python Traceback exception.
- **End-to-End Verification Pipeline**: 
  - Proves that the `.trace` format is successfully passed across boundaries. A single trace generated by Phase 2 can be reliably consumed by Phase 3, Phase 4, and Phase 5 without format errors or deserialization failures.

---

## 2. Phase 6 Architecture Deviations

1. **CLI Driven E2E Flow (No PyTest Orchestration)**
   - **Deviation**: The End-to-End test pipeline is executed manually (or via simple shell scripts) across consecutive CLI commands (`record`, `replay`, `inspect`, `eval`, `html-view`).
   - **Reasoning**: Instead of relying on a heavy testing framework like `pytest` to mock and wire together all five modules in memory, the E2E flow tests the actual compiled entry points that a user will interact with.

---

## 3. Known v1 Tradeoffs (By Design)

1. **No Automated Incident Report Generator**
   - **Compromise**: If an E2E run fails, the developer must manually inspect the generated `.trace` file or HTML output to see what went wrong.
   - **Reasoning**: Advanced automated incident reporting and failure classifications (e.g., auto-categorizing a failure as an "Auth Error" vs "Hallucination") were explicitly deferred to v1.1.
2. **Single Unbranched Execution**
   - **Compromise**: The E2E pipeline processes paths linearly. It starts at Step 1 and ends at Step N based on the initial recording.
   - **Reasoning**: Lacking any "what if" or branched re-execution capabilities in this version, the E2E suite only guarantees playback reliability, not dynamic resilience testing.
3. **Security/Permission Engine Bypass Clarification (`cli/record.py`)**
   - **Compromise**: In `cli/record.py`, there are direct `open("test.txt", "w")` and `sqlite3.connect(db_path)` calls that do not route through `security/permission_engine.py`.
   - **Reasoning**: These are test-fixture/demo-data setup calls used to artificially seed the environment before the actual tool execution is recorded. They are entirely outside the recorded execution path of the simulated agent and are NOT a threat model violation or bypass of the Permission Engine.
4. **OpenEval Adapter Scenario Coverage Gap**
   - **Compromise**: While Acceptance Test 4 mandates "predefined benchmark scenarios" (plural), currently only a single `canonical-task` scenario is wired up.
   - **Reasoning**: This is a known gap in v1 testing scope; additional benchmark scenarios have not yet been implemented.
5. **Flattened Git History**
   - **Compromise**: Granular phase-by-phase commit history was not preserved for Phases 4–6. The work is squashed into `Initial commit` and `Phase 6 Fixes`.
   - **Reasoning**: This represents a traceability and audit-trail loss for these phases, but does not affect system correctness or security functionality.
