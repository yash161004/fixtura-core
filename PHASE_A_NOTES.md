# Phase A Notes: Trace Lineage Schema

- **Schema Design for Lineage Fields**: We opted to store `parent_trace_id` and `divergence_step_id` as a special `"event_type": "trace_header"` event written as the very first line of any new branch trace file. Old traces without this header are implicitly treated as root traces. This is backward-compatible and avoids creating a new file format just for metadata.
- **Trace ID Resolution (`resolve_trace_path`)**: Resolving a `parent_trace_id` to an actual file on disk during lineage resolution is currently wrapped in a seam `resolve_trace_path(trace_id: str, base_dir: Path) -> Path` inside `trace_reader.py`. Currently it assumes same-directory file lookup (e.g., `parent_path = base_dir / f"{trace_id}.trace"`). If we swap storage backends later, this single function is where we hook in the new resolution logic (e.g. DB lookup or remote fetch).
- **Validation Error Handling**: The recent update to `TRACE_FORMAT_SPEC.md` allowing `"validation_error"` in the `permission_decision` enum has been integrated directly into `TraceReader` validation logic so that branch traces can accurately capture and replay these validation outcomes.
