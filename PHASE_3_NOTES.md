# Phase 3 Notes: Replay Runtime

## Implementation Summary

- **TraceReader Validation (`recorder/trace_reader.py`)**: Validates every event directly upon read. Enforces uniqueness of `step_id`, validates `event_type` (`llm_call`, `tool_call`), and strictly enforces all unconditional and conditional field requirements from `TRACE_FORMAT_SPEC.md` (e.g. `permission_reason` is required iff `denied`, `response` is null iff `denied`). Any violation throws a fail-fast `TraceValidationError`.
- **PassiveReplay (`replay/passive_replay.py`)**: Safely yields recorded interactions sequentially. By design, no external modules, LLM providers, or the `PermissionEngine` are ever instantiated or queried during replay. Additionally, recorded tool executions with non-null `exception` fields explicitly reproduce their failures locally to prevent silent error swallowing.
- **StepInspector (`replay/step_inspector.py`)**: Provides a strictly read-only wrapper around `PassiveReplay` that pauses at every single step, advancing exclusively when `advance()` is explicitly invoked.

## Phase 2 Architecture Deviations

1. **Monotonic `step_id` Initialization**
   Originally, `ExecutionRecorder` implicitly trusted the `step_id` generated by the caller. To strictly satisfy uniqueness and prepare for collision-free sequences, `ExecutionRecorder` now internally manages a monotonic counter (`self._step_counter = 1`) and explicitly assigns/overwrites the `step_id` immediately prior to serialization (e.g., `step-000001`, `step-000002`). This is a structural change to an existing Phase 2 file.

## Known v1 Tradeoffs

1. **No Counterfactual or Branching Support**
   As stipulated in `ROADMAP.md`, no mechanisms for branching from a recorded trace or executing modified inputs are provided in this version. The Step Inspector provides strict read-only access to historical sequence states.
